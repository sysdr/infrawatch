# Automation UI System - Day 104

Production-grade workflow automation management with visual builder, script management, execution monitoring, and analytics dashboard.

## Quick Start

**Start (run from the project root `automation_ui_system/`):**
```bash
cd /path/to/automation_ui_system
./build.sh
```

Then open **http://localhost:3000** in your browser (not just "localhost"—you must include the port).

**Stop:**
```bash
/path/to/automation_ui_system/stop.sh
```

### "Connection refused" or "Can't reach this page" on localhost

- Use **http://localhost:3000** for the UI and **http://localhost:8000** for the API. Plain "localhost" (port 80) has no service.
- Start the stack first: from `automation_ui_system/` run `./build.sh` and wait until it prints "Open in your browser: http://localhost:3000".
- If using WSL and opening the browser in Windows, the script binds the frontend to `0.0.0.0` so it should be reachable; use `http://localhost:3000` (Windows localhost reaches WSL).
- If the frontend still fails to start, run `npm install` and `npm start` from `automation_ui_system/frontend/`, then open http://localhost:3000.

**Demo data (so dashboard is not all zeros):**
```bash
# With backend + Postgres running:
/path/to/automation_ui_system/scripts/run_demo.sh
```
Then open http://localhost:3000 → Dashboard. You should see Total Workflows ≥ 1, Success Rate &gt; 0, and Recent Executions.

```bash
PYTHONPATH=. pytest tests/ -v
```

## Duplicate Services

Before starting again, run `./stop.sh` to avoid duplicate backend/celery/frontend or Docker containers.

## Dashboard Validation

- **Total Workflows:** Created via Workflow Builder or seed script.
- **Success Rate / Failed / Avg Duration:** From workflow executions (run a workflow from Workflow Builder → Execute).
- **Execution Timeline / Status Distribution:** Populate by running workflows; seed script adds 3 successful executions.
- If values are zero, run `scripts/run_demo.sh` (with backend + DB up) or create and execute a workflow from the UI.
